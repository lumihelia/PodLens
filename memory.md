# Project Memory

## Current State
PodLens publishes bilingual (zh/en) podcast/paper interpretations to a static site in `docs/` (GitHub Pages, custom domain `lens.lumihelia.com`). Source markdown bodies live in `.podlens/episodes/`, manifest at `docs/manifest.json`, rendering logic in `podlens/publish.py`. Each episode is published via a one-off `scripts/publish_<name>.py` script that calls `publish_report()`.

The interpretation pipeline (`podlens/interpreter.py`) now runs on a **selectable provider** — `PODLENS_PROVIDER=gemini` (default) or `deepseek`, set in `.env`. Both are kept side by side (not a rip-and-replace) so cost/quality can be A/B'd and reverted with one env line. DeepSeek's small context window (~64K tokens) is handled by transparent chunking (`podlens/chunking.py`) for Stages 1-2 of the podcast pipeline only.

## Completed (continued)
- **2026-06-22**: Added DeepSeek as a second LLM provider, plus transcript chunking so long episodes still work on it.
  - **Provider switch, not a replacement**: `podlens/config.py::Config` gained `provider` (read from `PODLENS_PROVIDER`, default `gemini`); `has_api_key`/`missing_key_message` resolve to the right env var (`GEMINI_API_KEY` / `DEEPSEEK_API_KEY`) and default model (`gemini-2.5-pro` / `deepseek-chat`) per provider. Gemini code path is untouched — verified byte-identical call count (1 call per stage, no chunking notes) via a mocked-client test.
  - **Single SDK boundary**: `interpreter.py::_LLMClient` is now the ONLY place touching a provider SDK (`google.genai` or `openai.OpenAI` pointed at `https://api.deepseek.com`); every call site uses `.generate(model, prompt, temperature)`. Added `openai>=1.0.0` to `requirements.txt`.
  - **Chunking** (`podlens/chunking.py`, new): conservative token estimate (chars/1.5, deliberately overestimates so real usage stays under budget), `CONTEXT_BUDGET_TOKENS = {"deepseek": 50_000}` (no entry for gemini → chunking is a no-op for it). `split_transcript` splits on transcript LINE boundaries (never separates a timestamp from its text); falls back to word-level splitting for the rare single giant line with no breaks (verified both paths with synthetic transcripts).
  - **Map-reduce on Stages 1-2 only**: `interpreter.py::_reconstruct` and `_retell_plain_language` chunk the transcript, run the existing per-chunk prompt variants (`prompts.py::build_reconstruction_chunk_prompt` / `build_plain_language_chunk_prompt`), then ONE synthesis call merges partials into the same Markdown shape the unchunked path produces (`build_reconstruction_synthesis_prompt` / `build_plain_language_synthesis_prompt`). Stage 2's chunk budget subtracts the (already-merged, much smaller) reconstruction's token cost, so it re-chunks more finely than Stage 1 when needed. Stage 3 (insight + personal mapping) and `interpret_paper()` were deliberately NOT touched — Stage 3 only ever sees the condensed reconstruction + plain_language (small), and papers are normally well under the budget.
  - **Cost tradeoff, stated explicitly to Helia**: chunking turns 3 calls into roughly `2N + M + 2` (N = stage-1 chunks, M = stage-2 chunks) for a long episode — more calls at DeepSeek's lower per-call price. Net cost effect not yet measured against a real long episode; flagged as unverified.
  - Verified offline only (no real DeepSeek key yet — `.env` still has `DEEPSEEK_API_KEY=your_key_here`): both provider clients construct without error; `--dry-run` unaffected; chunking math and line-boundary integrity confirmed on synthetic 200K+ char transcripts (reassembly byte-equal to input); mocked-client test confirms call counts and progress-note sequencing for both the chunked and unchunked paths.
